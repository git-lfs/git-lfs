    <article class="markdown-body entry-content" itemprop="text"><h1><a id="user-content-git-lfs-specification" class="anchor" href="#git-lfs-specification" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Git LFS Specification</h1>
<p>This is a general guide for Git LFS clients.  Typically it should be
implemented by a command line <code>git-lfs</code> tool, but the details may be useful
for other tools.</p>
<h2><a id="user-content-the-pointer" class="anchor" href="#the-pointer" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>The Pointer</h2>
<p>The core Git LFS idea is that instead of writing large blobs to a Git repository,
only a pointer file is written.</p>
<ul>
<li>Pointer files are text files which MUST contain only UTF-8 characters.</li>
<li>Each line MUST be of the format <code>{key} {value}\n</code> (trailing unix newline).</li>
<li>Only a single space character between <code>{key}</code> and <code>{value}</code>.</li>
<li>Keys MUST only use the characters <code>[a-z] [0-9] . -</code>.</li>
<li>The first key is <em>always</em> <code>version</code>.</li>
<li>Lines of key/value pairs MUST be sorted alphabetically in ascending order
(with the exception of <code>version</code>, which is always first).</li>
<li>Values MUST NOT contain return or newline characters.</li>
<li>Pointer files MUST be stored in Git with their executable bit matching that
of the replaced file.</li>
</ul>
<p>The required keys are:</p>
<ul>
<li><code>version</code> is a URL that identifies the pointer file spec.  Parsers MUST use
simple string comparison on the version, without any URL parsing or
normalization.  It is case sensitive, and %-encoding is discouraged.</li>
<li><code>oid</code> tracks the unique object id for the file, prefixed by its hashing
method: <code>{hash-method}:{hash}</code>.  Currently, only <code>sha256</code> is supported.</li>
<li><code>size</code> is in bytes.</li>
</ul>
<p>Example of a v1 text pointer:</p>
<pre><code>version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345
(ending \n)
</code></pre>
<p>Blobs created with the pre-release version of the tool generated files with
a different version URL.  Git LFS can read these files, but writes them using
the version URL above.</p>
<pre><code>version https://hawser.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345
(ending \n)
</code></pre>
<p>For testing compliance of any tool generating its own pointer files, the
reference is this official Git LFS tool:</p>
<p><strong>NOTE:</strong> exact pointer command behavior TBD!</p>
<ul>
<li>
<p>Tools that parse and regenerate pointer files MUST preserve keys that they
don't know or care about.</p>
</li>
<li>
<p>Run the <code>pointer</code> command to generate a pointer file for the given local
file:</p>
<pre><code>$ git lfs pointer --file=path/to/file
Git LFS pointer for path/to/file:

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345
</code></pre>
</li>
<li>
<p>Run <code>pointer</code> to compare the blob OID of a pointer file built by Git LFS with
a pointer built by another tool.</p>
<ul>
<li>
<p>Write the other implementation's pointer to "other/pointer/file":</p>
<pre><code>$ git lfs pointer --file=path/to/file --pointer=other/pointer/file
Git LFS pointer for path/to/file:

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345

Blob OID: 60c8d8ab2adcf57a391163a7eeb0cdb8bf348e44

Pointer from other/pointer/file
version https://git-lfs.github.com/spec/v1
oid sha256 4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345

Blob OID: 08e593eeaa1b6032e971684825b4b60517e0638d

Pointers do not match
</code></pre>
</li>
<li>
<p>It can also read STDIN to get the other implementation's pointer:</p>
<pre><code>$ cat other/pointer/file | git lfs pointer --file=path/to/file --stdin
Git LFS pointer for path/to/file:

version https://git-lfs.github.com/spec/v1
oid sha256:4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345

Blob OID: 60c8d8ab2adcf57a391163a7eeb0cdb8bf348e44

Pointer from STDIN
version https://git-lfs.github.com/spec/v1
oid sha256 4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
size 12345

Blob OID: 08e593eeaa1b6032e971684825b4b60517e0638d

Pointers do not match
</code></pre>
</li>
</ul>
</li>
</ul>
<h2><a id="user-content-the-server" class="anchor" href="#the-server" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>The Server</h2>
<p>Git LFS needs a URL endpoint to talk to a remote server.  A Git repository
can have different Git LFS endpoints for different remotes.  Here is the list
of rules that Git LFS uses to determine a repository's Git LFS server:</p>
<ol>
<li>The <code>lfs.url</code> string.</li>
<li>The <code>remote.{name}.lfsurl</code> string.</li>
<li>Append <code>/info/lfs</code> to the remote URL.  Only works with HTTPS URLs.</li>
</ol>
<p>Git LFS runs two <code>git config</code> commands to build up the list of values that it
uses:</p>
<ol>
<li><code>git config -l -f .lfsconfig</code> - This file is checked into the repository and
can set defaults for every user that clones the repository. Note: Git LFS used
".gitconfig" instead of ".lfsconfig" until Git LFS v1.1. Git LFS will continue
to read ".gitconfig" if ".lfsconfig" does not exist until Git LFS v2.0.</li>
<li><code>git config -l</code> - A user's local git configuration can override any settings
from <code>.gitconfig</code>.</li>
</ol>
<p>Here's a sample Git config file with the optional remote and Git LFS
configuration options:</p>
<pre><code>[core]
  repositoryformatversion = 0
[lfs]
  url = "https://github.com/xaxamin/git-lfs.git/info/lfs"
[remote "origin"]
  url = https://github.com/xaxadmin/git-lfs
  fetch = +refs/heads/*:refs/remotes/origin/*
  lfsurl = "https://github.com/xaxadmin/git-lfs.git/info/lfs"
</code></pre>
<p>Git LFS uses <code>git credential</code> to fetch credentials for HTTPS requests.  Setup
a credential cache helper to save passwords for future users.</p>
<h2><a id="user-content-intercepting-git" class="anchor" href="#intercepting-git" aria-hidden="true"><svg aria-hidden="true" class="octicon octicon-link" height="16" version="1.1" viewBox="0 0 16 16" width="16"><path fill-rule="evenodd" d="M4 9h1v1H4c-1.5 0-3-1.69-3-3.5S2.55 3 4 3h4c1.45 0 3 1.69 3 3.5 0 1.41-.91 2.72-2 3.25V8.59c.58-.45 1-1.27 1-2.09C10 5.22 8.98 4 8 4H4c-.98 0-2 1.22-2 2.5S3 9 4 9zm9-3h-1v1h1c1 0 2 1.22 2 2.5S13.98 12 13 12H9c-.98 0-2-1.22-2-2.5 0-.83.42-1.64 1-2.09V6.25c-1.09.53-2 1.84-2 3.25C6 11.31 7.55 13 9 13h4c1.45 0 3-1.69 3-3.5S14.5 6 13 6z"></path></svg></a>Intercepting Git</h2>
<p>Git LFS uses the <code>clean</code> and <code>smudge</code> filters to decide which files use it.  The
global filters can be set up with <code>git lfs install</code>:</p>
<pre><code>$ git lfs install
</code></pre>
<p>These filters ensure that large files aren't written into the repository proper,
instead being stored locally at <code>.git/lfs/objects/{OID-PATH}</code> (where <code>{OID-PATH}</code>
is a sharded filepath of the form <code>OID[0:2]/OID[2:4]/OID</code>), synchronized with
the Git LFS server as necessary.  Here is a sample path to a file:</p>
<pre><code>.git/lfs/objects/4d/7a/4d7a214614ab2935c943f9e0ff69d22eadbb8f32b1258daaa5e2ca24d17e2393
</code></pre>
<p>The <code>clean</code> filter runs as files are added to repositories.  Git sends the
content of the file being added as STDIN, and expects the content to write
to Git as STDOUT.</p>
<ul>
<li>Stream binary content from STDIN to a temp file, while calculating its SHA-256
signature.</li>
<li>Atomically move the temp file to <code>.git/lfs/objects/{OID-PATH}</code> if it does not
exist, and the sha-256 signature of the contents matches the given OID.</li>
<li>Delete the temp file.</li>
<li>Write the pointer file to STDOUT.</li>
</ul>
<p>Note that the <code>clean</code> filter does not push the file to the server.  Use the
<code>git push</code> command to do that (lfs files are pushed before commits in a pre-push hook).</p>
<p>The <code>smudge</code> filter runs as files are being checked out from the Git repository
to the working directory.  Git sends the content of the Git blob as STDIN, and
expects the content to write to the working directory as STDOUT.</p>
<ul>
<li>Read 100 bytes.</li>
<li>If the content is ASCII and matches the pointer file format:
<ul>
<li>Look for the file in <code>.git/lfs/objects/{OID-PATH}</code>.</li>
<li>If it's not there, download it from the server.</li>
<li>Write its contents to STDOUT</li>
</ul>
</li>
<li>Otherwise, simply pass the STDIN out through STDOUT.</li>
</ul>
<p>The <code>.gitattributes</code> file controls when the filters run.  Here's a sample file that
runs all mp3 and zip files through Git LFS:</p>
<pre><code>$ cat .gitattributes
*.mp3 filter=lfs -text
*.zip filter=lfs -text
</code></pre>
<p>Use the <code>git lfs track</code> command to view and add to <code>.gitattributes</code>.</p>
